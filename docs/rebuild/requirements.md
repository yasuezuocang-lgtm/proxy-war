# proxy-war 要件定義

## 1. 機能要件

### 1.1 セッション開始

- F-01: 片側依頼人が Bot にDMを送るとセッションが作成される
- F-02: 同一ギルドでは同時に1セッションのみアクティブ
- F-03: セッション作成と同時に `AgentA` `AgentB` のインスタンスを準備する。相手側依頼人参加までは対応エージェントは待機状態

### 1.2 入力収集

- F-10: 依頼人は複数回に分けて本音を送れる
- F-11: 短時間連投は1入力として束ねる
- F-12: 入力は対応する側の `AgentMemory` の `privateBrief` に蓄積される
- F-13: 不足情報があれば `BriefComposer` が追加質問を返す
- F-14: 追加質問回数には上限を設ける（デフォルト3回）

### 1.3 要約確認

- F-20: `BriefComposer` が依頼人向け確認文を生成する
- F-21: 依頼人は承認または修正を返せる
- F-22: 修正時は既存ブリーフを更新し再要約
- F-23: 承認後、対応エージェントの `privateBrief` が確定する

### 1.4 ゴール設定

- F-30: 各依頼人は任意で「公開ゴール」を設定できる
- F-31: 公開ゴールは相手側エージェントから見える唯一のゴール表現
- F-32: 依頼人は別途「私的ゴール」も設定できる。これは自側エージェントのみ参照可
- F-33: ゴール未設定でも対話は開始できる

### 1.5 代理対話

- F-40: 両者の `AgentMemory` 確定後、`#talk` でターン制対話を開始
- F-41: 各ターンは `DebateCoordinator` が対象エージェントの `generateTurn` を呼んで実施
- F-42: エージェントは戻り値として `message` か `hearing` のいずれかを返す
- F-43: 1ラウンドの最大ターン数は設定値（デフォルト10ターン）
- F-44: A/B 同一エージェントの連続発言は禁止（公平性）

### 1.6 ヒアリング

- F-50: エージェントが `hearing` を返した場合、対応側依頼人へDMで質問送信
- F-51: 返答は対応エージェントの `absorbHearingAnswer` のみが取り込む
- F-52: ヒアリング回数は各側ごとに上限（デフォルト2回/ラウンド）
- F-53: タイムアウト時（デフォルト10分）は既存情報のまま対話再開

### 1.7 判定

- F-60: 最大ターン到達 or 両者降参で `JudgeAgent.judgeRound` を起動
- F-61: 判定入力に依頼人ブリーフ本文は含めない（公開ログ＋公開ゴール＋過去判定のみ）
- F-62: 判定は構造化 JSON で返り、最低限以下を含む
  - 勝者（A / B / Draw）
  - 採点項目とスコア
  - 総評
  - 落とし所
  - 振り返りの知見
- F-63: JSON が不正なら `JudgeAgent` は再試行→失敗時は引き分けにフォールバック

### 1.8 上告

- F-70: 引き分け以外では敗者のみ上告できる
- F-71: 上告可能時間は設定値（デフォルト24時間）
- F-72: 上告回数の上限は審級数で表現（地裁→高裁→最高裁の3審）
- F-73: 再審時は新しい `DebateRound` を追加（既存ラウンドは保持）
- F-74: 再審の `JudgeAgent` は新規インスタンスとして起動。過去判決は入力に含めるが、別の判断主体である

### 1.9 終了

- F-80: 上告がない or 最終審到達でセッション終了
- F-81: 必要に応じて敗者へ個別DMを送る
- F-82: 終了時にセッションをアーカイブ保存

### 1.10 リセット

- F-90: 依頼人が `リセット` 等のキーワードDMで現セッション破棄
- F-91: リセットは両側の `AgentMemory` を独立に破棄する
- F-92: 片側リセット要求でセッション全体が終了する

## 2. 非機能要件

### 2.1 完全分離原則（最優先）

- N-01: A エージェントは B のブリーフ本文を一度も参照しないこと（コンパイル時制約 + 実行時アサート）
- N-02: A 側発言生成と B 側発言生成は別インスタンスが担当すること
- N-03: `AgentMemory` は side ごとに独立し、相互参照を持たないこと
- N-04: 司会者と審判は LLM を直接呼ばないこと
- N-05: A 用と B 用のプロンプトは別エントリとして PromptCatalog に登録されること

### 2.2 保守性

- N-10: 状態遷移は `SessionStateMachine` 単一の場所で管理する
- N-11: Discord SDK 依存は `presentation` と `infrastructure/discord` のみ
- N-12: LLM 呼び出しは `LlmGateway` 抽象経由のみ

### 2.3 テスト性

- N-20: ドメイン層は LLM・Discord 不在でテストできる
- N-21: 各 `ParticipantAgent` は単体でモック相手にテストできる
- N-22: 司会者は LLM モック・エージェントモックで状態遷移を検証できる
- N-23: 情報リーク防止チェックリスト（agents.md §6）はテストで自動化する

### 2.4 安全性

- N-30: 依頼人入力は暗号化保存
- N-31: Bot は相手側ブリーフを表示・転送しない
- N-32: ログには privateBrief 本文を出力しない
- N-33: ヒアリング回答は対応側 `AgentMemory` のみに格納

### 2.5 運用性

- N-40: ログからセッション単位の流れを追える
- N-41: タイムアウト・LLM エラー・Discord API エラーの扱いを明示
- N-42: A/B どちらの API 呼び出しか識別可能（メトリクス・コスト分析向け）

## 3. 業務ルール

- R-01: 同一セッションで利用できる情報は自側 `privateBrief` のみ
- R-02: 相手の主張は公開対話ログ経由でのみ知る
- R-03: 上告中は敗者以外は上告できない
- R-04: 判定 JSON が不正な場合はフォールバック処理
- R-05: 依頼人未返信で停止しないようタイムアウト後の進行ルールを持つ
- R-06: 同一発話者の連続ターンを許容しない

## 4. 受け入れ条件

- A-01: セッションが開始から終了まで仕様通り遷移する
- A-02: 各状態で許可される操作と禁止される操作が明確
- A-03: 完全分離原則 N-01〜N-05 がすべて満たされる
- A-04: ヒアリング・判定・上告の例外系が定義されている
- A-05: Discord 依存コードを差し替えずにコアロジックがテストできる
- A-06: A/B の片側だけプロンプト・モデル変更しても他方に影響しない

## 5. スコープ外（明示）

- 文体・口調・人格設計は要件に含めない（実装層のプロンプト調整に閉じる）
- 多人数対戦
- Discord 以外の本番 UI
- 課金・分析基盤
- 音声入出力
- Web 管理画面

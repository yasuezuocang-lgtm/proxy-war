# proxy-war 再構築要件定義

## 1. 目的

- Discord 上で当事者同士が直接ぶつからずに、本音の整理、代理対話、判定、振り返りまでを行えるシステムを再構築する
- 現行の「動くプロトタイプ」から、仕様駆動で拡張可能なプロダクトへ移行する
- Discord 固有実装とコアロジックを分離し、将来の UI 差し替えを可能にする

## 2. 背景

- 現行実装は `src/bot/client.ts` に責務が集中している
- README 記載と実装仕様に差異がある
- 状態遷移、例外系、上告、ヒアリングの扱いがコード読解に依存している
- LLM プロンプトが重要仕様である一方、アプリケーション仕様として整理されていない

## 3. スコープ

### 3.1 今回作るもの

- セッション開始から終了までの一連フロー
- 本音入力、要約確認、ゴール設定、代理対話、ヒアリング、判定、上告、終了
- Discord を利用した DM と `#talk` チャンネルでの入出力
- 暗号化されたセッション保存
- LLM プロバイダー切り替え

### 3.2 今回やらないもの

- 音声入力
- Web 管理画面
- 多人数対戦
- Discord 以外の本番 UI
- 決済、課金、分析基盤

## 4. 対象ユーザー

- 対立やすれ違いを言語化したい当事者
- 当事者間の議論を安全に仲介したい運営者
- 今後システムを拡張する開発者

## 5. 用語定義

- 依頼人: Bot に DM で本音を送る実ユーザー
- 代理 Bot: 依頼人の情報のみを持って代理発言する Bot
- セッション: 1つの対立案件に対する一連のやり取り
- ブリーフ: 依頼人入力を LLM が構造化し、対話用に要約した情報
- ヒアリング: 代理対話中に依頼人へ追加確認する仕組み
- ラウンド: 1審級分の代理対話と判定のまとまり
- 上告: 敗者が次審へ進める操作

## 6. 体験要件

- 依頼人は DM だけで参加できる
- 依頼人は相手やサーバー管理者に本音全文を見られない
- Bot は依頼人の立場を取り違えずに要約する
- Bot は知らない事実を捏造しない
- 判定は結果だけでなく理由も返す
- 終了時に振り返りの材料が残る

## 7. 機能要件

### 7.1 セッション開始

- 片側の依頼人が Bot に DM を送るとセッションを開始できる
- 同一ギルド内では同時に 1 つのアクティブセッションのみ扱う
- セッション開始時に相手参加待ちの状態へ入る

### 7.2 入力収集

- 依頼人は複数回に分けて本音を送信できる
- 短時間の連投は 1 つの入力として束ねる
- LLM は入力を構造化し、不足情報があれば追加質問する
- 追加質問回数には上限を持つ

### 7.3 要約確認

- LLM は依頼人向け確認文を生成する
- 依頼人は承認または修正を返せる
- 修正が来た場合は既存分析を更新し再要約する
- 承認後に対話用システムプロンプトを確定する

### 7.4 ゴール設定

- 各依頼人は任意で勝ち取りたいゴールを設定できる
- ゴール未設定でも代理対話は開始できる

### 7.5 代理対話

- 両者準備完了後に `#talk` でターン制の代理対話を始める
- 各ターンは相手の直前発言に応答する
- 代理 Bot は自分側ブリーフにない事実を使ってはならない
- 代理 Bot は必要時のみヒアリングを要求できる
- 1 ラウンドの最大ターン数を設定できる

### 7.6 ヒアリング

- 代理 Bot は `[HEARING:質問]` 形式でヒアリングを要求できる
- ヒアリング対象の依頼人へ DM で質問を送る
- 返答があればブリーフを更新して対話を再開する
- タイムアウト時は既存情報のまま対話を再開する
- ヒアリング回数は各側ごとに上限を持つ

### 7.7 判定

- 審判 AI は対話全文、両者背景、ゴール、過去判決を入力に判定する
- 判定結果は構造化データで返す
- 判定には少なくとも以下を含める
- 勝者
- 採点項目
- スコア
- 総評
- 落とし所
- 振り返りの知見

### 7.8 上告

- 引き分け以外では敗者のみ上告できる
- 上告可能時間を設定できる
- 上告回数には上限を持つ
- 再審時は過去判決を保持しつつ新ラウンドを開始する

### 7.9 終了

- 上告がない、または最終審まで到達したらセッションを終了する
- 必要に応じて敗者へ個別メッセージを送る
- セッションをアーカイブ保存する

### 7.10 リセット

- 依頼人の DM によって現在セッションを破棄できる
- リセット後は新規セッションを開始できる

## 8. 非機能要件

### 8.1 保守性

- 状態遷移はユースケース外に散らさず、単一の状態機械で管理する
- Discord SDK 依存をコアロジックへ持ち込まない
- LLM 呼び出しは抽象化して差し替え可能にする

### 8.2 テスト性

- ドメイン層は LLM や Discord なしでテストできる
- 主要ユースケースはモックで単体テストできる
- 状態遷移は表形式またはシナリオテストで検証できる

### 8.3 安全性

- 依頼人入力は暗号化して保存する
- Bot は相手側のブリーフを参照してはならない
- Discord 側へ出る文面は必要最小限とする

### 8.4 運用性

- ログからセッション単位の流れを追える
- タイムアウト、LLM エラー、Discord API エラーの扱いを明示する

## 9. 業務ルール

- 同一セッション内で利用できる情報は自側依頼人のブリーフのみ
- 相手の主張は対話ログ経由でのみ知る
- 上告中は敗者以外は上告できない
- 判定 JSON が不正な場合はフォールバック処理へ遷移する
- 依頼人の未返信で止まり続けないよう、タイムアウト後の進行ルールを持つ

## 10. 主要ユースケース

### UC-01 依頼人が最初の本音を送る

- 前提: アクティブセッションがない
- 結果: セッション作成、入力受付開始

### UC-02 依頼人が追加情報を送る

- 前提: 入力中または要約確認中
- 結果: 構造化情報更新、必要なら再要約

### UC-03 依頼人が要約を承認する

- 前提: 要約確認中
- 結果: 対話用ブリーフ確定、準備完了へ進行

### UC-04 代理対話を開始する

- 前提: 両者が準備完了
- 結果: `#talk` に対話開始を通知しラウンド開始

### UC-05 代理 Bot がヒアリングを要求する

- 前提: 対話中かつ必要情報不足
- 結果: DM で確認、返答に応じて対話再開

### UC-06 審判 AI が判定する

- 前提: ラウンド対話終了
- 結果: 判定結果通知、必要なら上告待ちへ進行

### UC-07 敗者が上告する

- 前提: 上告可能状態
- 結果: 次審のラウンド開始

## 11. 受け入れ条件

- 仕様に沿ってセッションが開始から終了まで遷移する
- 各状態で許可される操作と禁止される操作が明確である
- 代理 Bot が相手側非公開情報へアクセスしない
- ヒアリング、判定、上告の例外系が定義されている
- Discord 依存コードを差し替えずにコアロジックをテストできる
